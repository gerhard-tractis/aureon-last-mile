## What Went Well

### 1. 100% Delivery on Doubled Scope
The Sprint Change Proposal expanded scope from 4 to 8 stories mid-sprint, adding full automation worker infrastructure. The team delivered all 8 stories without compromising quality.

### 2. Architectural Simplification Wins
Every time the team chose the simpler approach, it paid off:
- **Railway → Hostinger VPS ($6.99/month):** Eliminated managed service complexity and cost
- **Redis/BullMQ → PostgreSQL `FOR UPDATE SKIP LOCKED`:** Supabase's native locking was simpler and sufficient
- **OpenClaw/Groq LLM → plain Playwright + CSS selectors:** "The Beetrack UI is stable enterprise software — CSS selectors are sufficient. LLM browser agents add complexity with no benefit for a single-site scrape."

### 3. Adversarial Code Review as Primary Quality Gate
Every single story underwent adversarial code review, catching ~56 issues total. 20 HIGH priority issues were fixed before merge. This directly contributed to zero production incidents.

### 4. Live E2E Verification on Real Data
The defining practice of this epic. E2E testing on the VPS with real orders from Easy and Beetrack exposed spec-vs-reality gaps (especially in Story 2.5) that would never have been caught otherwise. This practice validated the entire pipeline end-to-end.

### 5. Packages Table Scope Expansion (Story 2.1)
Early domain insight that "packages are the scannable entities, not orders" led to adding the packages table during Story 2.1. This saved a future migration and unblocked Stories 2.5 and 2.6 which both needed dual order+package inserts.

### 6. Canonical Validation Module (Story 2.2)
Creating `orderImportValidation.ts` as the single source of truth for phone, date, and comuna validation established a reusable pattern across the epic.

## Challenges Encountered

### 1. Spec-vs-Reality Gaps (Most Pervasive)
**Impact:** Significant rework required in Stories 2.5 and 2.6

**Story 2.5 (Critical):**
- Entire CSV column mapping was wrong — actual Easy CSV headers differed completely from spec
- Data model assumption wrong: each CSV row = 1 carton/package, NOT 1 order
- `Entrega` was the real order number, not `Nro Carga` (load/manifest number)
- Required complete workflow rebuild from single-table insert to grouped dual-table upsert

**Story 2.6:**
- Original OpenClaw/Groq LLM approach scrapped entirely in favor of simpler Playwright + CSS selectors
- Office Online link resolution needed (Beetrack sends Office Online links, not direct downloads)

**Root Cause:** No real data samples available during planning. Specs were based on assumptions.

### 2. n8n Sandbox Limitations (Stories 2.5, 2.6)
Repeatedly hit by platform constraints discovered only during implementation:
- `fetch()` not available in Code nodes — must use HTTP Request nodes
- Binary data uses `filesystem-v2`, not base64 — must use native nodes (Compression, Extract from File)
- Execution retry replays with the workflow snapshot from original execution, not current version
- XLSX Extract with `headerRow: false` outputs `item.json.row` as arrays, not objects

### 3. TypeScript Typing Friction (Stories 2.2, 2.8)
- Supabase PostgREST generating `never` types — required `any` casts
- Zod `z.preprocess` producing `unknown` type inference — needed `z.transform().pipe()` pattern
- RHF + Zod type mismatches — required separate `FormInput` types (pre-transform)
- Missing `<Toaster />` from sonner in `app/app/layout.tsx` — all toasts would have been invisible

### 4. Silent Failure Modes (4/8 stories)
Found only through adversarial code review:
- Env var fallbacks silently providing empty strings (Story 2.1)
- Missing role-based access control on import page (Story 2.2)
- `crypto.ts` `decipher.update()` returning Buffer without encoding — silent data corruption (Story 2.6)
- Stale `todayISO` module-level constant never updating (Story 2.8)

### 5. n8n Workflows Untestable in CI
n8n workflow logic cannot be unit-tested in the CI pipeline. The team duplicated Code node logic in separate test files — acknowledged as fragile technical debt.

---

## Key Insights

### 1. Get Real Data Samples Before Writing Specs
Story 2.5's complete rebuild was avoidable. Future connector/integration stories must have real data samples attached before development begins.

### 2. Simplicity Wins — Choose Boring Technology
Every architectural simplification paid off. VPS over Railway, PostgreSQL over Redis, Playwright over LLM. When choosing between approaches, default to the simpler one.

### 3. Adversarial Code Review is Non-Negotiable
56 issues caught, zero production incidents. This is the primary quality gate and must continue.

### 4. E2E Verification on Real Infrastructure is the Gold Standard
This practice defined Epic 2's success and should be applied to future epics where applicable.
