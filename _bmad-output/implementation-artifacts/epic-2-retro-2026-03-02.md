## Challenges Encountered

### 1. Story 2.6: Multiple Architecture Pivots
**Impact:** Story 2.6 took the longest, requiring 3 major pivots

**Timeline:**
1. Started with Playwright browser connector (beetrack.ts)
2. Discovered direct API export approach, moved to n8n HTTP Request
3. API URL was wrong (`/dispatch_guides_list/search` → 404), had to reverse-engineer JS bundle to find correct endpoint (`GET /search`)
4. IMAP trigger conflict with Easy CSV workflow caused cross-contamination (Easy data processed as DispatchTrack)

**Root Cause:** DispatchTrack has no public API documentation. The export endpoint, CSRF requirements, and parameter format had to be discovered empirically.

### 2. IMAP Email Conflicts Between Workflows
**Impact:** Both Easy and Beetrack workflows shared the same IMAP inbox, stealing each other's emails

**Problem:** Two n8n IMAP triggers on the same mailbox (`contacto@transportesmusan.com`) with no IMAP-level filtering. Each trigger marked ALL unseen emails as read, preventing the other from processing them.

**Solution:** Added IMAP search criteria at the protocol level:
- Beetrack: `["UNSEEN", ["FROM", "dispatchtrack"]]`
- Easy CSV: `["UNSEEN", ["FROM", "easy.cl"]]`

**Lesson:** When multiple workflows share an inbox, ALWAYS filter at the IMAP level, not in post-processing code nodes.

### 3. n8n Sandbox Limitations
**Impact:** Multiple runtime errors during workflow development

**Examples:**
- `$env` access blocked by default (requires `N8N_BLOCK_ENV_ACCESS_IN_NODE=false`)
- `fetch()` not available in Code nodes (must use HTTP Request nodes)
- Binary data uses `filesystem-v2`, not base64 — can't process in Code nodes
- MCP partial updates created connections with `type: "0"` instead of `type: "main"`
- Execution retry replays with workflow snapshot from original execution, not current version

**Lesson:** n8n Code nodes are sandboxed. Always use native nodes for HTTP, binary, and file operations.

### 4. Empty DispatchTrack Exports
**Impact:** Confusion during testing — Mar 2 export was empty (headers only, no data rows)

**Root Cause:** DispatchTrack exports only contain completed deliveries. If no deliveries completed for a given date, the XLSX has headers but zero rows.

**Solution:** Map & Validate code throws `No valid orders from 0 rows` when XLSX is empty. Extract Spreadsheet returns 0 items, so the pipeline stops gracefully without errors.

## Key Insights

### 1. n8n Over Custom Workers
**Decision:** *"n8n workflows replaced the custom Node.js worker entirely for data ingestion. The visual editor, built-in IMAP/HTTP nodes, and scheduled triggers are far more maintainable."* - Charlie (Senior Dev)

- The `apps/worker` codebase (cron.ts, beetrack.ts) is now largely superseded by n8n
- n8n provides retry logic, error handling, and execution history out of the box
- Future connectors should be n8n workflows first

### 2. Reverse-Engineering Third-Party APIs is Costly
**Insight:** Story 2.6 took 3x longer than estimated because DispatchTrack has no documented API

- The correct export endpoint (`GET /search?excel_export=true&...`) was found by decompiling their React frontend bundle
- CSRF token requirement was discovered through trial-and-error
- Session cookie management adds fragility (cookies expire, need recalibration)

**Mitigation:** Document all discovered API endpoints and parameters. Monitor for API changes.

### 3. IMAP Triggers Need Protocol-Level Filtering
**Insight:** Code-level email filtering is too late — by then the IMAP trigger has already marked emails as read

- Always use `customEmailConfig` with IMAP search criteria
- Never share an IMAP credential between workflows without proper filtering
- Each workflow should only fetch emails it will process

### 4. Test with Real Data Early
**Insight:** Testing with real DispatchTrack data (not Easy CSV data) would have caught the column mapping and IMAP conflict issues days earlier

- The first "successful" test actually processed Easy data through the Beetrack pipeline
- Real DispatchTrack columns (`Orden`, `Nombre cliente`, `CARTONID`) differ completely from Easy columns (`Entrega`, `Nombre`, `Cartón`)
